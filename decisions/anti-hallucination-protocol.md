# Decision: Анти-галлюцинационный протокол

**Дата:** 2026-05-30
**Статус:** Принято
**Причина:** Инцидент с фейковым "Комплексным анализом системы"

---

## Что произошло

Был сгенерирован отчёт "Комплексный анализ системы: Объединённый отчёт" якобы по Zai-agent-toolkit. Отчёт содержал:

- `transaction_processor.js`, `transaction_handler.js` — файлы которые **не существуют**
- `user_model.js`, `user_repository.js` — файлы с расширением `.js` — в репо **нет ни одного .js файла**
- `auth_service.py`, `user_controller.py` — Python-файлы с типичными бэкенд-именами, **не существуют**
- `api_client.js`, `api_server.js`, `cache_manager.js`, `data_service.js` — **ни одного нет**

**Реальность:** 259 файлов в репо, 35 skills, 19 standards, 7 tools, 20 dashboard .tsx, 10 API routes. Ни один файл из отчёта не совпал с реальностью. **Точность: 0%.**

Отчёт анализировал **типичный Express/FastAPI проект** — стандартный AI-шаблон, не имеющий отношения к реальному тулкиту.

---

## Почему это произошло

1. Отчёт был запрошен без предоставления исходных файлов
2. AI сгенерировал "правдоподобный" анализ по шаблону серверного проекта
3. Не было обязательной проверки: "существует ли файл, на который я ссылаюсь?"
4. Результат выглядел профессионально — но был полностью выдуман

---

## Правило: не читал — не утверждаешь

**Любое утверждение о файле, функции или структуре проекта должно быть подтверждено чтением реального файла.**

| Действие | Обязательно | Как проверить |
|----------|------------|---------------|
| Упомянуть файл в отчёте | Да | `find . -name "файл"` — должен найтись |
| Указать на баг в коде | Да | Прочитать файл, увидеть проблему своими глазами |
| Описать архитектуру | Да | `find . -type f` → прочитать ключевые файлы |
| Предложить фикс | Да | Понять текущий код, не предполагать |
| Сказать "готово" | Да | `git diff` должен показать изменения |

---

## Протокол перед любым анализом

```
Шаг 1: git clone (или git pull) — получить актуальный код
Шаг 2: find . -type f — получить полный список файлов
Шаг 3: Прочитать каждый файл, на который будешь ссылаться
Шаг 4: Каждое утверждение проверить по прочитанному
Шаг 5: Если файл не найден — НЕ придумывать. Написать: "файл не найден"
```

---

## Красные флаги галлюцинации

| Флаг | Что это значит | Что делать |
|------|---------------|-----------|
| Упомянут `.js` файл в TypeScript проекте | AI подставил шаблон | Проверить `find . -name "*.js"` |
| Имена вроде `*_service`, `*_controller`, `*_model` | Типичный MVC-шаблон | Проверить существует ли |
| Нет конкретных путей к файлам | Анализ общими словами | Запросить конкретику |
| Нет конкретных номеров строк | "Где-то там" = нигде | Читать файл, указывать строки |
| Красивое оформление, но 0 проверяемых фактов | Фасад без содержания | Возврат к Шагу 1 |

---

## Что делать с галлюцинированным отчётом

**Выбросить. Не исправлять. Не дополнять.**

Галлюцинированный отчёт не содержит верных фактов — исправлять нечего. Правильный анализ начинается с чтения реальных файлов, а не с правки выдуманных.

---

## Цена ошибки

Этот фейковый отчёт отнял ~1 неделю работы. Неделя на:
- Попытки понять "найденные" проблемы
- Проверку несуществующих файлов
- Написание других программ для проверки ошибок
- Фрустрацию и потерю доверия

**Одна галлюцинация = одна потерянная неделя. Поэтому протокол — не рекомендация, а правило.**
