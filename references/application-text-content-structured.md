### Проблема

LLM не имеют постоянной памяти (stateless). Каждый запрос — с чистого листа. Отправка всей истории сжигает токены и упирается в лимит контекстного окна.

## 3. 4 основные техники

**Подзаголовок:** От простого к сложному — выберите подходящую стратегию

### Техника 1: Суммаризация
- **Подзаголовок:** Сжимаем историю
- **Описание:** Вместо полной истории отправляем краткий пересказ
- **Экономия:** 70-90%
- **Сложность:** Низкая
- **Лучше для:** Быстрая экономия в длинных чатах

### Техника 2: Иерархическая память
- **Подзаголовок:** Краткосрочная + Долгосрочная
- **Описание:** Делим память на «оперативку» и «жёсткий диск»
- **Экономия:** 60-80%
- **Сложность:** Средняя
- **Лучше для:** Продвинутые ассистенты

### Техника 3: Векторные базы (RAG)
- **Подзаголовок:** Ищем релевантное
- **Описание:** Эмбеддинги + векторный поиск по истории
- **Экономия:** 80-95%
- **Сложность:** Высокая
- **Лучше для:** Огромные истории и документация

### Техника 4: Извлечение фактов
- **Подзаголовок:** Профиль пользователя
- **Описание:** Храним структурированные данные вместо переписки
- **Экономия:** 90-99%
- **Сложность:** Средняя
- **Лучше для:** Долгосрочная персонализация

## 4. Сравнение расхода токенов

**Подзаголовок:** Для диалога из 50 сообщений (~8000 токенов истории)

| Метод | Токены | Описание | Экономия |
|-------|--------|----------|----------|
| Без управления памятью | 8 000 | Полная история + системный промпт | — |
| Суммаризация | 1 500 | Системный промпт + саммари | -81% |
| Иерархическая | 1 200 | Системный промпт + саммари + последние 5 msg | -85% |
| RAG | 800 | Системный промпт + релевантные чанки | -90% |
| Извлечение фактов | 300 | Системный промпт + JSON профиль | -96% |

### Ключевые инсайты

**Максимальная экономия:** Извлечение фактов сокращает расход токенов на 96% — с 8 000 до 300 токенов на запрос.

**Качество памяти:** RAG и извлечение фактов сохраняют детали, которые суммаризация теряет. Идеально для персонализации и сложных ассистентов.

**Стоимость контекста:** Без управления памятью, 50 сообщений стоят ~$0.08/запрос. С оптимизацией — от $0.003 до $0.015 за запрос.

**Совет:** Комбинируйте техники: суммаризация для общей картины + RAG для точных деталей. Это даёт баланс экономии и качества.

#### 5.3 RAG — Векторный поиск (RAGDemo)

**Сценарии:** Поддержка Nginx | E-commerce поддержка | Медицинский помощник | Юридический ассистент

**Сценарий: Поддержка Nginx**
- Запрос: «Как настроить кэширование в Nginx?»
- Релевантные чанки: proxy_cache_path, location /static/ expires 30d, reverse proxy upstream
- Нерелевантные: SSL, 502 Bad Gateway, gzip, MySQL

**Сценарий: E-commerce поддержка**
- Запрос: «Как оформить возврат товара?»
- Релевантные: Возврат через ЛК → Мои заказы, 14 дней, обмен за 7 дней, возврат 3-5 дней
- Нерелевантные: Доставка, промокоды, оплата, лояльность

**Пайплайн RAG:**
1. Разбиваем историю на чанки
2. Превращаем в вектор (embedding) — 1536 измерений
3. Сохраняем в векторную БД (Pinecone, Chroma, Qdrant)
4. При новом запросе ищем похожие (top_k=3)
5. Вставляем найденное в промпт

**Плюсы:** Очень экономно, помнит детали из любой давности, масштабируется на миллионы.

**Минусы:** Требует векторную БД, нужна модель эмбеддингов, зависит от размера чанков.

**Описание:** Практические способы экономить токены для пользователей ChatGPT/Claude.

| Цель | Стратегия |
|------|-----------|
| Экономия денег | Суммаризация + Новый чат |
| Долгосрочный проект | Custom Instructions + Ветвление |

**Совет 1: Custom Instructions** — Впишите контекст в настройки аккаунта. Модель всегда «знает» кто вы.

**Совет 3: GPTs с Knowledge Base** — Прикрепите файлы до 512MB, модель сама ищет через RAG.

**Совет 4: Ветвление диалогов** — Создавайте «ветки» для разных тем вместо одного длинного чата.

**Совет 5: System Prompt вместо истории** — Поместите контекст в системный промпт (10-20 токенов).

**Совет 6: Чёткий вопрос** — Чем точнее вопрос, тем меньше контекста нужно модели.

#### 5.14 Архитектура (ArchitecturePipeline)

**Уровни:**

**Простой** — Базовая архитектура без управления памятью
- Пользователь → Отправляет запрос → Получает полную историю → Ответ
- Никакая информация не сохраняется между запросами

**С суммаризацией** — Добавлено сжатие истории
- Менеджер памяти проверяет лимит → Сборка контекста (промпт + саммари + последние) → Ответ
- Обновление: суммаризация при превышении лимита

**Production** — Полная архитектура с вектором, фактами и иерархией
- Оркестратор: RAG-поиск + обновление профиля + пересуммаризация
- Векторная БД: эмбеддинги, быстрый семантический поиск
- Хранилище фактов: JSON-профиль пользователя
- Минимальный, но полный контекст. Качество на уровне человеческой памяти.

**Пример потока данных:**
«Помнишь, какой фреймворк я использую?» → RAG-поиск → fact_store.get(«фреймворк») → Контекст: «Пользователь: Мария, React + TypeScript» → «Да, вы работаете с React и TypeScript!»

#### 5.23 Глоссарий (Glossary)

| Термин | Определение | Сложность |
|--------|-------------|-----------|
| Токен (Token) | Минимальная единица текста для LLM. ~4 символа (EN), ~2-3 (RU). GPT-4: 128K. | Базовый |
| Контекстное окно | Макс. объём текста за запрос. GPT-4: 128K, Claude 3.5: 200K. | Базовый |
| Эмбеддинг | Числовое представление текста. Похожие смыслы = близкие векторы. | Средний |
| Векторная БД | БД для хранения и поиска векторов (Pinecone, Chroma, Qdrant). | Средний |

| Суммаризация | Сжатие длинной истории в краткое резюме. 20 сообщений → 1 абзац. | Базовый |
| Sliding Window | Хранятся только последние N сообщений. Одна строка кода. | Базовый |
| Извлечение фактов | Структурированные данные из переписки (JSON/БД). Контекст не растёт годами. | Средний |
| Промпт | Текстовый запрос: системный (инструкции) + сообщения пользователя. | Базовый |
| Системный промпт | Инструкция для LLM: роль, стиль, ограничения. Не растёт. | Средний |

#### 5.27 Кейсы (RealWorldCaseStudies)

**Case 1: ChatGPT / Claude.ai** — Суммаризация + Sliding Window
- GPT-4: 128K контекст, ~$10/1M tok, сжатие 60-80%
- Плюсы: Простота, большое окно, автоуправление
- Минусы: Информация за окном теряется, задержка при сжатии

**Case 2: MemGPT / Letta** — Иерархическая память
- RAM: ~8K токенов, Архив: ∞, semantic поиск
- Плюсы: Неограниченная память, LLM сам управляет, Open-source
- Минусы: Сложная настройка, дополнительные API-вызовы

**Case 3: Notion AI / Perplexity** — RAG + Fact Extraction
- 1000+ документов, 5-10 чанков/запрос, поиск ~50мс
- Плюсы: Масштабируется, экономия, источники цитируются
- Минусы: Требует векторную БД, зависит от чанкинга

**Case 4: Cortex / Character.ai** — Fact Extraction + Иерархическая
- ~50 фактов о пользователе, хранение ∞, персонализация ~95%
- Плюсы: Глубокая персонализация, структурированные факты
- Минусы: Приватность, устаревшие факты

## 8. Главный вывод

Не существует одной «лучшей» техники управления памятью. Правильный подход — **комбинировать методы** в зависимости от сценария. Начните с суммаризации для быстрой экономии, затем добавьте иерархическую память для продвинутых use-кейсов, и интегрируйте RAG или извлечение фактов для production-систем.

- Простые чаты → Суммаризация
- Ассистенты → Иерархическая
- Production → RAG + Факты
